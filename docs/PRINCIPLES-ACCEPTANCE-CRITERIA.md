## AC-P1 收费时点与结果导向（P0 —— 已定案落地）

> 定案（2026-07-21）：**预付 = 托管冻结；验收成功后才计入平台收入；取消 / 争议退款 = 服务费托管自动全额退回。**

| ID | 验收标准 | 判定 |
|----|----------|------|
| AC-P1.1 | 明确产品方案并落地：服务费预付写入 `WalletServiceFeeHold`（HOLD），验收 `consume` 后才计入平台收入 | 云端代码 + 表结构 |
| AC-P1.2 | 客户端「取消订单」相关文案写明自动全额退回，不再使用「按规则退还」 | 文案审查 |
| AC-P1.3 | `prepay` / `cancel` / `confirm` / 管理端争议裁决与托管语义一致；旧 debit 预付路径有 credit 回退兼容 | 代码审查 |
| AC-P1.4 | 预付确认页 / 钱包 / 帮助中心显式说明「托管非消费」 | UI 文案审查 |

## AC-P2 收费透明度（P0）

> 对照原则："所有收费内容必须公开、清楚，不同用户不应看到不同的规则。"

| ID | 验收标准 | 判定 |
|----|----------|------|
| AC-P2.1 | 客户端所有展示费率的位置（按钮文案、钱包页顶、Wallet 详情）均读取服务端返回的真实 `serviceFeeRate` 字段，删除所有硬编码「5%」字符串 | 全文检索无硬编码百分号费率字符串 |
| AC-P2.2 | 服务端字段缺失时，UI 展示「费率以下单结算为准」而不是回落默认值（0.05）伪装成真实数据 | 代码审查：无 `?? 0.05` 类回退 |
| AC-P2.3 | 结算详情 / 钱包流水详情展示「成交价 × 费率 = 手续费」计算过程（需服务端在 DTO 补 `feeRate`、`baseAmount` 字段配合） | DTO + UI 联调 |
| AC-P2.4 | 《服务费规则》《服务协议》从纯文字染色改为可点击跳转的真实文档/页面 | UI + 导航 |
| AC-P2.5 | 帮助中心「托管费用如何计算」FAQ 填入真实说明（比例、计算公式、结算时间） | 文案内容审查 |
| AC-P2.6 | 设计预览 fixture 中的费率数字（当前 10%）与线上真实费率保持一致，避免 QA/产品误判 | 代码审查 |

## AC-P3 去紧迫感 UI（P1）

> 对照原则：不制造停留/焦虑的心理钩子，即便动机是业务时效性。

| ID | 验收标准 | 判定 |
|----|----------|------|
| AC-P3.1 | `DiscoverView` / `DemandDetailView` 的「剩余可见」展示去掉红色强调色与倒计时跳动效果，改为中性文字（如「预计失效于 HH:mm」） | UI 审查 |
| AC-P3.2 | `CardPoolView` 排序选项保留「即将截止」作为客观排序维度，但列表项展示不使用倒计时式紧迫感样式 | UI 审查 |
| AC-P3.3 | `.urgent` 态字段可以保留用于业务逻辑，但不强制驱动视觉警示色 | 代码审查 |

---

## AC-P4 推送默认值与分类别开关（P1）

> 对照原则：推送完全交给用户自定义，不能"默认帮你订阅好了"。

| ID | 验收标准 | 判定 |
|----|----------|------|
| AC-P4.1 | 新用户首次设置页默认值改为关闭，或在首次进入时显式询问用户是否开启匹配推送（不能静默默认开） | 代码 + 首次登录流程审查 |
| AC-P4.2 | 站内通知按大类（订单/需求/系统/福利）提供独立开关，不再只有一个粗粒度总开关 | UI + Service 联调（需服务端 `/pushes/preferences` 支持分类字段） |
| AC-P4.3 | 设置页暴露 `excludeRegions` 字段的编辑入口，与已有的排除关键词/标签并列 | UI |
| AC-P4.4 | 认证通过后是否自动开启"带标签推送"需与云端确认；若是自动订阅，改为显式询问 | 云端确认 + 客户端联调 |

## AC-P6 本轮明确不做（避免范围膨胀）

- 会员制/差别化服务：本轮审计未发现问题，不安排修复任务
- 付费曝光/中介化：本轮审计未发现问题，不安排修复任务
- Loop 状态轮询机制调整：属正常进度同步，非沉迷设计，不动
- 圈子「置顶公告」（免费功能）：不动，仅记录风险防止未来误加商业化

---

## 本轮修复顺序

1. AC-P1 收费时点决策（需产品+服务端联合拍板，最先启动因为周期最长）
2. AC-P2 收费透明度（客户端可独立推进的部分先做：AC-P2.1/2.2/2.6；依赖服务端字段的部分排后）
3. AC-P3 去紧迫感 UI（纯客户端，可并行推进）
4. AC-P4 推送默认值与分类别开关（客户端默认值先改，分类别开关等服务端字段）
5. AC-P5 演示数据清理（上线前收尾检查项）
