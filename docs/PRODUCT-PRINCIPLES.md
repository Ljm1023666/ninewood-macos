# 九木 — 产品理念北极星

> **性质：** 非工程文档，是决策文档。每次做产品/交互/商业化决策时，先对照本文判断"符不符合我们说过的话"。  
> **确立日期：** 2026-07-21  
> **变更方式：** 本文可以被讨论和修订，但修订应当是慎重的、经过讨论的，不应该为了某个短期指标而悄悄让步。

## 一、我们为什么做这件事

我们不希望做出像短视频平台那样让人上瘾的产品。那不是产品，是一种精心设计的消耗品，它偷走使用者的时间和注意力，让人忽视生活本身的美好。

九木存在的理由是相反的：**把被琐事、被信息壁垒、被算法绑架的时间，还给用户。** 产品不是必需品，但有它在，生活总要好一些。做到极致，是用完即走——用户尽快达成目的、尽快离开，是产品成功的信号，不是失败的信号。

需求和服务直接对接，中间没有中介、没有信息战。

- **天回** — 极致的匹配效率
- **地回** — 优秀顺滑的交互
- **人回** — 信任的延续（历史沉淀，不必每次重新博弈）

这三者共同节省时间，同时提高合作的可能性与成交率。

最终目标：人只为自己而活。每个人都值得拥有美好的一生。

## 二、可执行的产品原则

### 1. 反沉迷 / 反停留时长

- 不以"打开时长""日活分钟数""连续登录天数"作为内部成功指标。
- 不做无限滚动、不做"猜你喜欢"式的信息流填充，不用算法制造停留理由。
- 任务完成后的默认路径是"退出"，不是"继续看点别的"。
- 评估任何新功能时先问：这个功能是在帮用户更快达成目的，还是在制造多留一会儿的理由？

### 2. 推送完全交给用户自定义

- 推送不是运营工具，不是为了提升打开率或商业指标而存在的手段。
- 用户主动订阅什么、订阅到什么粒度，才决定会收到什么；没有"默认帮你订阅好了"的暗箱操作。
- 系统内部不应该存在"提升推送点击率"这类以平台利益为目标的指标。
- 不用推送给用户制造信息茧房；用户看到什么，是用户自己决定的，不是算法替他决定的。

### 3. 去中介化 / 去信息战

- 需求和服务之间，平台只做"连接"与"信任沉淀"，不做信息壁垒。
- 不做付费置顶、不做流量分级曝光——曝光机会不应该用钱购买。
- 信任基于真实的历史记录沉淀（人回），不是基于付费等级或人为包装。

### 4. 无差别服务

- 不做会员制，产品本身不应该设置门槛。
- 不做内购分层，每一个用户都应该获得完整的服务，不应该被区别对待。
- 代码层面：不应该出现 `if isPremium { ... }` 这种按付费等级分支功能可用性的逻辑。

### 5. 收费：单一入口、结果导向、完全公开

- 利润坍缩为单一入口：佣金 / 手续费，不做数字广告、不做内购、不做会员费。
- 结果导向：没有帮到用户，不产生任何费用；帮到了，才收取极低比例的手续费维持平台运转。
- 这要求系统内部有一套**清晰、可审计的"成交/完成"判定逻辑**（不能模糊地"默认收费"）。
- 所有收费内容必须公开、清楚，不含糊其辞，不同用户不应看到不同的规则。
- 未来收费模式可以变得更灵活，但"完全公开、结果导向、单一入口"这三条是不可退让的底线。

## 三、决策自检清单

做任何新功能、任何交互、任何商业化改动之前，问这几个问题：

1. 这个设计是在替用户省时间，还是在替平台占用户的时间？
2. 这个通知/推送，是用户自己要的，还是我们觉得"这样能提升数据"才加的？
3. 这个曝光机会，是不是可以用钱买到？如果是，删掉它。
4. 这个功能，付费用户和不付费用户看到的是否一样？如果不一样，为什么？
5. 这笔费用，用户在事前是否能看懂、看全？如果看不懂，重新设计费率展示。
6. 如果我们这次没有帮到用户，这笔单子是否真的不产生费用？系统里有没有明确的判定逻辑来保证这一点？

只要有一条自检不过关，这个设计就需要重新讨论，而不是先上线再说。

## 四、一句话

我们希望做的，不是占据用户生活的产品，而是在用户需要的时候真正有用、用完之后就悄悄退到不远处的产品——如果之后还需要，我们一直在那儿。
