# macOS 架构现代化方案

> 目标：把“视图直接调用 API 的移植客户端”演进为可测试、可追踪、可持续扩展的原生客户端。  
> 原则：按业务风险迁移，不重写云端，不改变未经确认的资金规则。

## 验收矩阵

| 工作包 | 验收标准 | 状态 |
|---|---|---|
| 发布领域 | 校验、规范化、命令组装独立测试；View 不组装 API 参数 | 已完成 |
| Repository 边界 | 需求、订单、消息、自然回具备领域数据入口 | 已完成 |
| 主列表 Feature Model | 发现、订单、消息的加载/筛选/选中/错误离开 View | 已完成 |

| Swift 6 并发基线 | Natural Loop 默认参数隔离警告消除 | 已完成 |
| 领域测试入口 | 无需启动 UI 即可运行核心规则测试 | 已完成 |

## 实施约束

### 不改变的边界

- 生产 API 仍为云端权威；
- 不在客户端重算真实押金、退款或最终结算；
- 不修改用户现有 Agent 和云迁移脚本；
- 不用大规模文件移动掩盖功能变化；
- 未确认前不对生产数据执行发布、支付、结算或删除。

### 新代码规范

1. 页面状态进入 `Features/<Domain>/*FeatureModel.swift`。
2. 业务数据入口进入 `Data/Repositories`。
3. 可独立判断的规则进入 `Domain` 并提供测试。
4. 路由字符串只出现在 `API/Services`。
5. DTO 不进入 View；Repository 或 Mapper 转成领域模型。
6. 资金类按钮由领域策略决定是否出现，服务端仍做最终鉴权。

## 后续扩展顺序

1. 支付、部分结算、争议等独立 Sheet 的表单模型；
2. 申请人管理、Circle、Profile、Wallet 等次高风险页面；
3. Loop、Circle、Wallet DTO 分域及云端响应 Fixture；
4. 云端提供资金规则版本和明确金额字段后，移除客户端兼容推导；
5. 在测试账号和隔离钱包中执行完整 E2E。

这些迁移应保持“每迁一个页面即可构建、测试、人工验证”，禁止长时间维护不可运行的
大分支。
